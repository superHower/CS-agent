# CS-Agent 智能客服系统

## 项目概述

CS-Agent 是一个**多平台智能客服系统**，通过影刀 RPA 机器人接入淘宝、拼多多、京东、抖音四大电商平台，实现自动化的买家消息处理与回复。系统采用分层架构，集成了 FAQ 缓存、向量检索（RAG）、LLM 生成和意图识别等 AI 能力。

---

## 需求与业务场景

### 核心需求

1. **多平台消息接入**：支持淘宝、拼多多、京东、抖音四大电商平台的买家消息
2. **智能回复生成**：基于知识库的自动回复，支持 FAQ 精确匹配和 RAG 增强生成
3. **转人工机制**：通过硬关键词、置信度阈值等多维度判断是否需要人工介入
4. **管理后台**：配置店铺、FAQ、知识库、告警规则等
5. **记忆回写**：将会话记录异步写入 Obsidian 笔记库

### 技术选型理由

| 组件 | 选型 | 理由 |
|------|------|------|
| 消息接入 | 影刀 RPA | 无需申请平台 API，通过 UI 自动化抓取消息 |
| 向量检索 | Qdrant | 高性能向量数据库，支持实时检索 |
| 嵌入模型 | BGE-small-zh | 轻量高效的中文嵌入模型 |
| LLM | OpenAI 兼容接口 | 支持云端（GPT）和本地（Ollama）灵活切换 |
| 缓存 | Redis | FAQ 缓存、会话存储、消息去重 |
| 知识库 | Obsidian | Markdown 格式，易于维护和检索 |
| 后端框架 | FastAPI | 异步高性能，支持 OpenAPI 自动文档 |
| 管理后台 | React Admin | 快速构建数据管理界面 |

## 核心模块详解

### 1. 网关层 (Gateway)

#### RpaGateway

负责接收影刀 RPA 机器人推送的消息，解析后投入调度队列。

**接口规范：**

```json
// POST /api/message
// 请求体 (新格式)
{
    "shop_id": "tb_lamp_001",
    "history": [{
        "platform": "淘宝",
        "shop": "艾睿斯旗舰店",
        "buyer": "买家昵称",
        "product": "商品名或无",
        "chatList": ["气泡1", "气泡2"],
        "detail": "订单详情或无"
    }],
    "message_id": "唯一ID"
}

// 响应
{
    "reply": "回复文本",
    "escalated": false  // true 表示已转人工
}
```

**关键特性：**
- 消息幂等去重 (Redis, 60秒窗口)
- 同步等待调度层回复 (Future 机制, 30秒超时)
- 支持新格式 (history) 和旧格式 (buyer_id + content) 兼容
- 分类信息从店铺配置获取，不再从消息传入

### 2. 调度层 (Scheduler)

#### SessionScheduler

7 分支状态机，实现消息处理的完整流程：

```
┌──────────────────────────────────────────────────────────────┐
│                     消息进入调度器                              │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│ 分支1: 加载/新建会话上下文 (Redis session:{shop}:{buyer})     │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│ 分支2: 会话状态检查 (WAITING_HUMAN → 安抚话术 + 转人工)        │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│ 分支3: 硬转人工关键词检查 (最高优先级)                          │
│        命中 → 搪塞话术 + 标记会话 + 告警                        │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│ 分支4-5: MatchEngine 决策 (FAQ/意图/RAG)                       │
│        置信度 ≥ 阈值 → 自动回复                                 │
│        置信度 < 阈值 → 转人工                                    │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│ 分支6: 发送回复 + 保存会话 + 更新置信度                         │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│ 分支7: 异步记忆回写 (Obsidian)                                │
└──────────────────────────────────────────────────────────────┘
```

### 3. 匹配层 (Matching)

#### MatchEngine

三步管道决策引擎（分类从店铺配置获取）：

```
┌──────────────────────────────────────────────────────────────┐
│ Step 1: FAQ 精确缓存命中                                      │
│         优先店铺专属 FAQ                                      │
│         未命中 → 分类共享 FAQ                                 │
│         Redis key:                                          │
│         - faq:shop:{shop_id}:{hash}                        │
│         - faq:category:{category_id}:{hash}                 │
│         命中 → 直接返回 (置信度 100%)                          │
└──────────────────────────────────────────────────────────────┘
                              │ 未命中
                              ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 2: LLM 意图识别 + 查询改写                               │
│         识别意图: 物流/售后/投诉/产品咨询/安装/推荐/闲聊/其他    │
│         改写查询: 更精准的检索词                                │
│         超时 2s → 降级用原查询                                 │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 3: Qdrant 向量检索 + LLM 生成                            │
│         优先店铺专属知识 → 分类共享知识                          │
│         Top-5 相关知识片段                                     │
│         合并上下文 + 调用 LLM                                  │
│         解析置信度标记 [CONFIDENCE: XX]                        │
│         总超时 7s → 转人工兜底                                │
└──────────────────────────────────────────────────────────────┘
```

#### IntentHandlers

| 意图类型 | 处理器 | 置信度调整 | 特殊处理 |
|---------|--------|-----------|---------|
| LOGISTICS | LogisticsHandler | 0 | 注入订单信息 |
| AFTER_SALE | AfterSaleHandler | -0.05 | 稍宽松 |
| COMPLAINT | ComplaintHandler | -0.1 | <0.6 直接转人工 |
| PRODUCT_INQUIRY | ProductInquiryHandler | 0 | 注入商品信息 |
| INSTALL_GUIDE | InstallGuideHandler | 0 | 标准 RAG |
| RECOMMEND | RecommendHandler | 0 | 注入商品信息 |
| CHITCHAT | ChitchatHandler | -0.1 | 简短回复 |
| OTHER | OtherHandler | 0 | 标准 RAG |

### 4. 知识检索层 (Retrieval)

#### 分层检索策略

```
Level 1: 店铺专属 FAQ 精确缓存 (Redis)
    key: faq:shop:{shop_id}:{hash}
    ↓ 未命中
Level 2: 分类共享 FAQ 精确缓存 (Redis)
    key: faq:category:{category_id}:{hash}
    仅当 category_id != "default" 时生效
    ↓ 未命中
Level 3: 快捷短语关键词 (ShortcutPhraseIndex)
    ↓ 未命中
Level 4: Qdrant 向量语义检索 (Top-5)
    优先店铺专属知识 → 分类共享知识
    有 tags/backlinks 加权 1.1x
```

#### FAQ 与知识管理模型

系统支持**分类级**和**店铺级**两层次的知识管理：

| 类型 | shop_id | category_id | 说明 |
|------|---------|-------------|------|
| 店铺专属 | 具体ID | 任意 | 仅该店铺可用 |
| 分类共享 | global | 具体ID | 该分类下所有店铺共享 |

**FAQ 示例**：
- 店铺专属 FAQ：`shop_id=tb_lamp_001`, `category_id=lamp`
- 分类共享 FAQ：`shop_id=global`, `category_id=lamp` → 所有灯饰类店铺可用

**Qdrant Collection**：
- `collection_{shop_id}`: 店铺专属知识层
- `collection_{category_id}`: 分类共享知识层

**检索优先级**：店铺专属知识 > 分类共享知识（匹配分数相同时，店铺专属优先）

### 5. LLM 层

#### 后端选择

| 配置 | CloudLLMBackend | LocalLLMBackend |
|------|-----------------|-----------------|
| backend | "cloud" | "local" |
| base_url | OpenAI 兼容接口 | http://localhost:11434 |
| 适用场景 | 生产环境 | 开发/测试 |

#### 置信度解析

LLM 输出格式：`回复内容 [CONFIDENCE: 85]`

### 6. 动作层 (Actions)

#### send_message.py

平台消息发送路由，支持重试 (2次, 1s 间隔)。

#### alert_human.py

企业微信机器人 Markdown 告警：

```markdown
**客服转人工通知**
>店铺：tb_lamp_001
>买家：买家昵称
>平台：taobao
>原因：置信度不足
>触发消息：xxx
>时间：2024-01-01 12:00:00 UTC
>买家：你好
>客服：亲，稍等
```

#### writeback.py

异步写入 Obsidian Markdown：

```markdown
# 买家昵称

## 2024-01-01
- 咨询：发货时间是多久；回复：一般3-5个工作日 → 已解决
  - 意图：产品咨询
  - 相关：[[发货政策]]
```

## 数据流与存储

### Redis 存储

| Key 模式 | 类型 | 用途 |
|---------|------|------|
| `session:{shop}:{buyer}` | Hash | 会话上下文 (history, state, confidence) |
| `faq:{shop}:{question_hash}` | String | FAQ 缓存，TTL 0 |
| `msg_dedup:{shop}:{msg_id}` | String | 消息去重，TTL 86400s |
| `config_updated` | Pub/Sub | 配置热更新通知 |

### SQLite 数据库

| 表 | 用途 |
|---|------|
| shops | 店铺配置 |
| categories | 分类配置 |
| faq_items + faq_aliases | FAQ 主表 + 别名 |
| products | 产品型号 |
| knowledge_entries | 知识条目 |
| knowledge_files | MD 文件管理 |
| escalation_keywords | 告警关键词 |
| decoy_phrases_pool | 搪塞话术 |
| message_logs | 消息日志 |
| conversation_archive | 会话归档 |
| daily_stats | 每日统计 |
| llm_config / alert_config | 配置表 |

## 部署与运行

### 依赖服务

| 服务 | 端口 | 用途 |
|------|------|------|
| Redis | 6379 | 缓存/会话 |
| Qdrant | 6333 | 向量检索 |
| 本地 LLM (可选) | 11434 | Ollama API |

### 启动方式

```bash
# 激活环境
conda activate knowledge_qa

# 启动主服务 (8080 端口)
python -m src.main

# 或使用 Makefile
make dev-server
```

### 启动流程

1. 初始化 SQLite 表
2. 加载配置 (YAML + SQLite)
3. 连接 Redis / Qdrant
4. 启动配置热更新监听
5. 初始化 FAQ 缓存、检索器、LLM 客户端
6. 启动后台任务：回写服务、调度器、各店铺 RPA 监听
7. 监听 HTTP 8080

## 安全考虑

1. **敏感词脱敏**: 买家昵称、手机号等写入 Obsidian 前脱敏
2. **配置隔离**: API Key 通过环境变量或数据库管理，不写入代码
3. **消息去重**: 60秒窗口防止重复处理
4. **超时保护**: 各层设置超时阈值，失败自动转人工
