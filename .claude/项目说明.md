# CS-Agent 项目说明 — `/api/message` 主链路详解

> 本文档聚焦**影刀 RPA 调用 `POST /api/message` 之后，系统内部发生了什么、最终又返回了什么**。所有平台差异、后台管理、数据库表结构等不在此展开。

---

## 1. 整体架构（一句话版）

`/api/message` 收到 RPA 推送的聊天气泡 JSON → 网关解析气泡、抽取"买家最新一句" → 标准化后投入调度队列 → 调度器走「会话上下文 → 硬关键词拦截 → 匹配决策引擎（FAQ → 意图识别 → 向量检索 → LLM 生成 → 置信度判定）」 → 决策结果要么直接回复，要么转人工（搪塞话术 + 告警） → 回复内容通过 `asyncio.Future` 同步回填到 HTTP 响应体里返回给 RPA。

整条链路的关键设计：
- **同步 HTTP、超时 30 秒**：RPA 是 HTTP 调用，必须在超时内返回；调度内部用 `asyncio.Future` 把"异步处理"转成"同步等待"。
- **多层降级**：任何一个环节挂了（LLM 超时、向量检索超时、解析失败），都不会让买家收到烂回复，宁可转人工。
- **Redis 为核心状态存储**：会话上下文、FAQ 缓存、去重、转人工时间戳都在 Redis。

## 2. 接口契约：请求与响应

### 2.1 请求体（`RpaMessageRequest`）

定义在 `src/gateway/rpa.py`，支持三种格式（按优先级自动识别）：

1. **平铺格式**（前端调试页 / MessageTest.tsx 常用）：
   ```json
   {
     "platform": "抖音",
     "shop": "艾睿斯旗舰店",
     "buyer": "张三",
     "kefu": "小李",
     "product": "无",
     "chatList": ["气泡1", "气泡2", "..."],
     "detail": "无"
   }
   ```
2. **新格式**（影刀 RPA 真实推送）：`{"shop_id": "tb_lamp_001", "history": [...]}`，`history[-1]` 是当前会话。
3. **旧格式**（兼容）：`{"shop_id": "...", "buyer_id": "...", "content": "...", "platform": "taobao"}`。

可选字段 `last_interaction_at`：chatList 最后一条气泡的 ISO8601 时间戳（UTC），用来在人工处理中判断"买家和客服还在平台上互动"。

### 2.2 响应体（`RpaMessageDebugResponse`）

```json
{
  "reply": "亲，36W 适合 9㎡左右的客厅…",
  "escalated": false,
  "shop_id": "tb_lamp_001",
  "shop_name": "艾睿斯旗舰店",
  "extracted_buyer": "张三",
  "extracted_message": "客厅用哪款合适",
  "history_turns_count": 4,
  "product_name": "...",
  "steps": [ ... 调试步骤 ... ],
  "final_source": "intent_rag",          // faq_cache | intent_rag | fallback
  "final_reply": "...",
  "confidence": 87,
  "confidence_threshold": 85,
  "total_elapsed_ms": 1234,
  "error": ""
}
```

`reply` 是真正要发给买家的文本，`escalated=true` 表示已转人工（reply 一般是搪塞话术）。

## 3. 端到端处理流程

下面按调用顺序把 `/api/message` 走过的每个环节展开。

### 3.0 路由入口（`src/main.py:202`）

`api_message()` 路由做的事：

1. **店铺解析**：优先用 `body.shop_id`；若空，按 `body.shop` 走 `/shops/resolve-name` 在管理后台 DB 里查 `shop_id` 与 `category_id`。
2. **动态启动 listener**：如果该 `shop_id` 还没有后台监听协程（`_run_gateway_listener`），就临时起一个 `_run_gateway_listener(rpa_gateway, shop_cfg, scheduler)`，往内部 `_queues[shop_id]` 灌消息。
3. **核心调用**：`await gw._handle_message(body)` — 这步是同步阻塞等待回复（最多 30 秒）。
4. **响应构造**：从 `_handle_message` 拿到的 `DebugContext` 里抽 `steps / confidence / source`，再从请求体里重新解析 `extracted_message / history_turns_count`，拼成调试响应返回。

注意 `_handle_message` 和 listener 是两条独立路径：
- **HTTP 路径**（`/api/message`）：`_handle_message` 同步等回复，未来再走异步任务也会把 reply 回填到同一个 Future。
- **Listener 路径**（后台监听）：每店铺一个协程从 `_queues[shop_id]` 取出消息，扔给 `scheduler.enqueue(msg)`，由 `SessionScheduler.run()` 的消费者处理。

两条路最终都汇到 `scheduler.dispatch()`，行为一致。

---

### 3.1 网关解析气泡 → 提取买家最新消息（`src/gateway/rpa.py:296` `_handle_message`）

进入 `_handle_message` 后，三种格式分支会分别做这些事（核心动作一致）：

1. **平台归一**：`_PLATFORM_MAP` / `_CN_PLATFORM_MAP` 把 `taobao / 淘宝 / 拼多多 / 抖音` 等映射到 `Platform` 枚举。
2. **店铺 ID 解析**：若 `shop_id` 为空，HTTP 调用 `/shops/resolve-name` 拿（`shop_id, category_id`）。
3. **气泡预处理**（抖音专属）：`filter_douyin_bubbles(bubbles, kefu)` 过滤掉系统消息气泡（关键词列表在 `src/gateway/rpa_parser.py:_DOUYIN_SYSTEM_KEYWORDS`），例如 "系统消息"、"退款成功"、"机器人接待中" 等。
4. **角色分类**（`_classify_bubble`）：按正则判断每个气泡是 `user / assistant / system`：
   - 含 "智能客服"、"客服XX发送"、"运营接入" 等 → `assistant`
   - 含 "已读" 结尾、"¥\d" 商品卡、"柔光护眼" 等 → `assistant`
   - 含 "系统关闭会话" 等 → `system`（丢弃）
   - 其它默认 → `user`
5. **气泡清洗**（`_clean_bubble`）：去掉独立的时间戳行（"昨天 12:29"）、"已读 / 未读" 标记，拼接成纯文本。
6. **提取最新买家消息**（`extract_latest_buyer_message`）：反向遍历已分类的 turns，找到最后一个 `role == "user"` 的 turn，返回其清洗后的内容。
7. **提取历史 turns**（`extract_history_turns`）：从最新一条 user 之前的所有 user/assistant 对话，取最近 6 轮。
8. **消息 ID 生成 / 去重**：
   - `message_id` 缺省时按 `shop_id:buyer_id:content:分钟数` MD5 生成。
   - `_is_duplicate()` 用 Redis `SETNX` 做 60 秒级别的去重，命中直接返回空 reply。
9. **构造 `StandardMessage`**（契约见 `src/contracts/models.py:96`）：打包 `shop_id / platform / buyer_id / content / timestamp / message_id / source=RPA / product_name / order_detail / raw_payload / raw_chat_list / kefu / chat_list_latest_at`。
10. **创建 `asyncio.Future` 与 `DebugContext`**：
    - `self._pending_replies[message_id] = future` — 调度层处理完后会 `future.set_result(reply)`。
    - `self._escalated[message_id] = False` — 标记是否转人工。
    - `self._pending_debug_ctx[message_id] = DebugContext()` — 后续每一步会把 `step / label / hit / reply / error / intent / chunks / elapsed_ms` 追加进去。
11. **入队**：`self._queues[shop_id].put(msg)`。
12. **同步等待**：`await asyncio.wait_for(future, timeout=30.0)`，超时则返回 `reply=""、escalated=True`（转人工兜底）。

这一步结束时，`StandardMessage` 已经躺在 `_queues[shop_id]` 里等消费，且 HTTP 调用被挂起等待 Future。

---

### 3.2 调度器入口（`src/scheduler/dispatcher.py:215` `_handle`）

Listener 协程 `await scheduler.enqueue(msg)` 后，`SessionScheduler.run()` 会 `asyncio.create_task(self._handle(msg))`，进入 `_handle`：

1. **解析店铺 / 分类**（`resolve_shop_info`）：从 DB 拿 `category_id`。
2. **加载 ShopConfig**：先查静态配置（`config.get_shop`），找不到就构造一个最小 `ShopConfig`（`confidence_threshold=85`，`category_id="default"`）。若静态配置存在但 DB 里有 `category_id`，以 DB 为准覆盖。
3. **调用 `dispatch(msg, shop_config)`** — 这就是 7 分支状态机。

### 3.3 分支 1：会话上下文加载 / 人工处理去抖（`dispatch:274`）

- `_store.load_or_create(msg)` → Redis 读 `session:{shop_id}:{buyer_id}`：
  - 有：反序列化 `SessionContext`、续期 TTL（默认 2h）、返回。
  - 无：新建 `SessionContext(state=ACTIVE, created_at, updated_at)` 返回。
- 若 `ctx.state == WAITING_HUMAN`，进入 **10 分钟去抖判断**：
  - 取 `max(handoff_at, msg.chat_list_latest_at, ctx.updated_at)` 作为锚点。
  - 现在 − 锚点 < 600 秒 → **静默忽略**（不发回复、不发告警，只把 `Future` 填上空 + escalated=true 让 HTTP 正常返回）。
  - 现在 − 锚点 ≥ 600 秒 → 视为新一轮对话，状态重置为 ACTIVE、清 `handoff_at`，继续往下走。

### 3.4 分支 2：订单/物流意图（`dispatch:327`）

`_is_order_query(content)` 仅做占位（订单系统模块未接入时跳过），**当前不会改变流程**。保留接口供后续接入。

### 3.5 分支 3：硬转人工关键词检查（`dispatch:332`）

`_get_escalation_keywords(shop_config)` 取关键词列表，优先级：
1. 动态加载（`load_dynamic_config` 启动时从 `escalation_keywords` 表查，按 `category_id+shop_id` 缓存）。
2. 实时 DB 查 `load_escalation_keywords(cat_id, shop_id)`。
3. 静态 YAML 配置 `config.escalation_keywords`。

`_check_hard_keywords(content, keywords)` 是子串包含匹配，命中第一个关键词就返回该词。命中后 `_do_escalate(reason=HARD_KEYWORD, triggered_keyword=...)`：发送搪塞话术 → 标 WAITING_HUMAN → 写 `handoff_at` → 调 `alert_service.notify(...)`。

### 3.6 分支 4 & 5：MatchEngine 决策（`dispatch:349`）

把上下文打包成 `MatchRequest`，交给 `MatchEngine.match(shop_config, req)`。整个 match 流程外面套了一个 7 秒的 `asyncio.wait_for`，超时则 fallback 转人工。

#### MatchEngine 内部三步管道（`src/matching/engine.py:122`）

##### Step 1：FAQ 精确缓存 + 快捷短语（`engine.py:160` + `src/retrieval/retriever.py:160`）

`MatchEngine.match → self._retriever.retrieve(shop_config, faq_query)` → `Retriever.retrieve` 按四层顺序检索：

1. **店铺专属 FAQ**（Redis）：`faq:shop:{shop_id}:{md5(question)}` — 命中即返回。
2. **分类共享 FAQ**：`faq:category:{category_id}:{md5(question)}`。
3. **快捷短语关键词匹配**（`ShortcutPhraseIndex`）：加载 `.claude/快捷短语.json`，按 `code` 长度降序做子串匹配，命中随机返回一条 `phrase`。这一步模拟"老客服常用的固定话术"。
4. **向量语义检索**（Qdrant）：
   - `QueryEnhancer.enhance(question)`：先做型号缩写展开（`config/product_dict.yaml`）再做否定句改写（如 "不亮" → "故障 不亮 灯不亮"），返回查询列表。
   - 拿主查询，调用嵌入模型（本地 SentenceTransformer 或 API 百炼 `text-embedding-v3`）。
   - **双层 collection**：`collection_{shop_id}` + `collection_{category_id}`（分类共享），各取 Top5、按 `score` 合并去重。
   - 含 `tags/backlinks` 的结果 `score *= 1.1`（加权），最后再按 `score` 降序取 Top5。
   - 整个向量检索有 300ms 硬超时，超时则返回空 chunks。

回 `MatchEngine` 后：
- 若 FAQ / 快捷短语命中 → 写 `step="faq_cache"` 调试步骤 → 直接返回 `MatchResult(reply=faq_reply, source="faq_cache", confidence=100)`。
- 若向量检索返回 chunks → 写 `step="rag"` 调试步骤（带 chunks_preview 和每条 score）。
- 若没 chunks → 写 `step="rag" hit=False`。

##### Step 2：LLM 意图识别（`engine.py:237`）

`build_intent_request()` 构造扩展上下文（抖音模式额外追加【商品名称】【订单详情】【买卖双方对话记录】），调 `_recognize_intent`：

1. **构造 Prompt**（`src/matching/intent_prompt.py`）：system prompt 是灯具电商专用的"12 类一级意图 + 风险等级 + 安抚话术规则"大段说明，强调"必须综合【买卖双方对话记录】整体判断，而不仅是最后一句"。
2. **调用 LLM**：**独立超时 2 秒**（`_INTENT_TIMEOUT_S = 2`），低温度（0.1）、短 max_tokens（200）。底层走 `aiohttp` 直接打 `/chat/completions`（不走 `LLMClient`，因为意图识别对延迟极敏感）。
3. **解析 JSON 响应**：从原文里 `find("{")` 到 `rfind("}")` 切片，`json.loads` 后填 `risk_level`（缺失默认 low）、`needs_escalation`（缺失按 risk_level 推算）。
4. **降级策略**：超时 / 异常 / 解析失败 / 无 JSON 段 → 返回 `IntentResult(intent="other", risk_level="high", needs_escalation=True, escalate_reason="意图识别失败/解析失败/格式异常")` — 设计上是**宁转人工也别乱答**。

`_recognize_intent` 完后做映射：`IntentType(intent_result.intent.lower())`，未知字符串降级为 `IntentType.OTHER`。`query = intent_result.rewrite_query or request.user_msg` — `rewrite_query` 是 LLM 给出的"更适配语义搜索"改写。

写调试步骤 `step="intent"`，记录 `intent / sub_intent / risk / entities / rewrite_query / elapsed_ms / error`。

##### Step 2.4：风险拦截（`engine.py:282`）

```
is_high_risk   = risk_level == "high" or needs_escalation
is_appeasement = intent_type in {PLATFORM_RISK, HEALTH_RISK}
```

- **非安抚类 + 高风险**（维权投诉 / 安全事故 / 12315 / 平台已介入 / 识别失败 等）→ 直接返回 `MatchResult(reply="", source="intent_xxx_risk_blocked", needs_escalation=True)`，**不再走 RAG / 生成**，交给上层 `_do_escalate` 转人工。
- **安抚类**（PLATFORM_RISK / HEALTH_RISK，加微信 / 健康不适等）→ 继续走 Step 2.5 和 Step 3，让 handler 生成一条安抚话术 + needs_escalation=true 转人工。

##### Step 2.5：改写查询二次检索（`engine.py:301`）

当原始检索无 chunks 且 `rewrite_query != user_msg` 时，再用 LLM 改写后的查询重跑一次向量检索（500ms 超时），结果合并入 `retrieval`。

##### Step 3：按意图路由到 handler（`engine.py:324`）

`dispatch_intent(intent_type, request, shop_config)` 查 `_INTENT_HANDLERS` 注册表（`src/matching/intent_handlers.py:389`），调用对应 handler。共有 12 个：

| IntentType | Handler | 风险 | 调整 |
|---|---|---|---|
| PRODUCT_INQUIRY | ProductInquiryHandler | low | 注入商品名称到 extra_knowledge |
| RECOMMEND | RecommendHandler | low | 同上 |
| LOGISTICS | AfterSaleLike | mid | `confidence_adjustment=-0.05`，注入订单详情+商品名 |
| INSTALL_GUIDE | InstallGuideHandler | mid | 标准 RAG |
| AFTER_SALE | AfterSaleHandler | mid | `confidence_adjustment=-0.05`，注入订单+商品名 |
| PRICE_PROMO | PricePromoHandler | mid | `confidence_adjustment=-0.05`，注入商品名 |
| COMPLAINT | ComplaintHandler | low | `confidence_adjustment=-0.1`，且 confidence < 0.6 强制转人工 |
| BULK_ORDER | BulkOrderHandler | mid | 注入商品名 |
| **PLATFORM_RISK** | PlatformRiskHandler | **high** | 注入"安抚话术 prompt"到 extra_knowledge，is_appeasement=True |
| **HEALTH_RISK** | HealthRiskHandler | **high** | 同上 |
| CHITCHAT | ChitchatHandler | low | `confidence_adjustment=-0.1` |
| OTHER | OtherHandler | mid | 标准 RAG |

Handler 内部实际统一调用 `MatchEngine._generate_with_context`（`engine.py:507`），逻辑：

1. 合并 `request.knowledge`（向量检索 chunks）+ `extra_knowledge`（订单/商品/安抚规则）作为 `knowledge_text`。
2. 取 `request.history[-6:]` 构造 `TurnRecord` 列表。
3. 构造 `LLMRequest(shop_id, shop_name, buyer_message, history, knowledge)`，调 `self._llm.generate(llm_req)`。
4. `LLMClient.generate` 走云端/本地后端（`_build_backend` 按 `cfg.backend` 选），整体 `asyncio.wait_for` 30 秒超时（可被 `cfg.timeout` 覆盖），超时抛 `LLMTimeoutException`。
5. 后端调用 OpenAI Chat 接口 → 真实大模型（如豆包 / GPT / Qwen）按 `prompt.py:_PROMPT_TEMPLATE` 生成回复（系统提示词含售后原则 + 灯具核心知识 + 检索片段 + "回复末尾附 [CONFIDENCE: XX]"）。
6. `LLMResponse.raw_text` 解析：抽 `reply` 文本 + `[CONFIDENCE: N]` 中的 N 作为 confidence。
7. `adjusted_confidence = clamp(response.confidence + confidence_adjustment, 0, 1)`。
8. `needs_escalation = adjusted_confidence < shop_config.confidence_threshold / 100`（安抚模式下强制 False，避免安抚话术因"不够准确"被转人工）。

写调试步骤 `step="llm"`（含 reply、confidence、knowledge_chars、elapsed_ms）。

最后 `_match_inner` 返回 `MatchResult(reply, source="intent_xxx", confidence=int(c*100), needs_escalation, intent=intent_type.value)`。

### 3.7 分支 4 收尾：决策分流（`dispatch:372`）

MatchEngine 返回后：

- `match_result.needs_escalation == True` → `_do_escalate(reason=LOW_CONFIDENCE, confidence=...)`。
- 否则：判断 `is_greeting`（短消息 + 含寒暄模式词）→ 寒暄也算通过。
- `confidence >= shop_config.confidence_threshold`（或寒暄且 confidence > 0）→ `_reply_and_save(ctx, msg, shop_config, reply, confidence)`：
  - `_send(shop_config, buyer_id, reply, {"message_id": msg.message_id})` → 实际是 `send_reply(gateway_registry, ...)` → `rpa_gateway.send(...)` → 把 reply 填回 `_pending_replies[message_id]` 这个 Future。
  - 发送成功 → 把本轮 user + assistant 加入 `ctx.history`（截断保留最近 10 轮）、更新 `last_confidence`、`_store.save(ctx)`。
  - `asyncio.create_task(_async_writeback(...))` 异步做记忆回写（当前实现只写日志，未真正落库）。
- 置信度不足 → `_do_escalate(reason=LOW_CONFIDENCE, ...)`。

### 3.8 转人工兜底（`dispatch:505` `_do_escalate`）

任何分支触发 `_do_escalate(msg, shop_config, reason, ctx, triggered_keyword, confidence)`：

1. **`_get_decoy_phrase(shop_config)`**：从 `decoy_phrases_pool` 表按 `category_id + shop_id` 取一条随机搪塞话术（如 "亲，稍等我查一下哈~"），`_send` 发给买家。
2. 构造 `EscalationContext{shop_id, buyer_id, platform, reason, trigger_message, recent_history, confidence, triggered_keyword, message_id, timestamp}`。
3. 更新会话 `state=WAITING_HUMAN`、`_store.save(ctx)`、写 `handoff_at`。
4. `_escalate(escalation)` → `main.py` 注入的 `escalate_fn`：
   - 通过 `rpa_gateway.send(metadata={message_id, escalated:True})` 标记买家侧已转人工。
   - 调 `alert_service.notify(ctx)`（`AlertService.from_config(config.alert)`）走钉钉 / 飞书 / webhook 等渠道告警。
5. 异常路径（dispatch 顶层 try/except）也会 `_do_escalate(reason=EXCEPTION)`。

### 3.9 同步回填与 HTTP 返回（`src/gateway/rpa.py:454`）

回到最早 `_handle_message` 的 `await asyncio.wait_for(future, 30s)`：

- 在 30 秒内，`scheduler.dispatch → _send → rpa_gateway.send` 会 `future.set_result(reply)`，并把 `_escalated[message_id] = escalated`。
- `_handle_message` 解包 `(reply, escalated)`，从 `_pending_replies / _escalated / _pending_debug_ctx` 弹出该 message_id（debug context 不清，留给路由层取），返回 `(RpaMessageResponse, message_id)`。
- 路由层 `api_message()` 取 `_pending_debug_ctx[message_id]` 构造调试响应，最终 JSON 返回。
- **超时**：future 没人填 → 抛 `TimeoutError` → 返回 `reply=""、escalated=True`（RPA 端识别到 escalated=true 就不发回复，告警通道触发人工接入）。

## 5. 调试响应里的 `steps` 字段

`RpaMessageDebugResponse.steps` 是前端 `MessageTest.tsx` 用来逐步展示链路执行的卡片数据。`api_message()` 路由从 `_pending_debug_ctx[message_id]` 读出，每步是一个 `RpaMessageDebugStep`：

| step | 触发位置 | 含义 |
|---|---|---|
| `faq_cache` | MatchEngine Step1 FAQ 命中 | 命中即返回，reply 就是 faq_reply |
| `rag` | MatchEngine Step1 向量检索 | 列出 chunks + score，无论有没有都记 |
| `intent` | MatchEngine Step2 LLM 意图识别 | 记录 intent/sub_intent/risk/entities/rewrite_query/elapsed_ms |
| `llm` | MatchEngine Step3 handler 生成 | 记录 reply/confidence/knowledge_chars |

`final_source` 由最后一步推断：
- `faq_cache` → `faq_cache`
- `llm` → `intent_rag`
- `intent` + `hit=False` → `fallback`（高风险拦截）

`confidence` 取最后一个 step 的 confidence；`confidence_threshold` 当前实现未填（恒为 None）；`total_elapsed_ms` 取所有 step 中最大值。

## 6. 异常与降级路径汇总

| 异常点 | 降级行为 |
|---|---|
| RPA 请求体格式异常（缺 shop_id 等） | `_handle_message` 抛 `ValueError` → HTTP 400 |
| Redis SETNX 失败（去重） | 跳过去重，正常处理 |
| Redis 会话读取失败 | 降级为新建会话 |
| FAQ 缓存读取失败 | 跳过该层，继续下一层 |
| 向量检索超时（300ms） | 返回空 chunks，由 MatchEngine Step2 继续 |
| LLM 意图识别超时（2s） | `IntentResult(intent="other", risk_level="high", needs_escalation=True)` |
| LLM 意图识别结果无 JSON 段 | 同上 |
| LLM 生成超时（30s） | `_do_escalate(reason=EXCEPTION)` |
| 意图 handler 抛异常 | 降级为标准生成 `_generate_with_context` |
| `match_result.needs_escalation=True` | `_do_escalate(reason=LOW_CONFIDENCE)` |
| `_send` 失败（Future 未填） | `_reply_and_save` 内检测 → 转 `SEND_FAILED` 人工 |
| `_handle_message` Future 30s 超时 | 返回 `reply=""、escalated=True`（兜底转人工） |
| dispatch 顶层异常 | `_do_escalate(reason=EXCEPTION)` |
| 买家在人工处理中 10 分钟内重复发 | 静默忽略，Future 填 `""`+`escalated=true` 保持 HTTP 正常返回 |

## 8. 容易被忽略的设计细节

1. **`is_appeasement=True` 时不强制转人工**：`_generate_with_context` 在该模式下 `needs_escalation = False`，即使 confidence 低也允许安抚话术发出；handler 之后再单独置 `needs_escalation=True`，确保上层转人工路径仍走，但买家已收到安抚。
2. **`chat_list_latest_at` 字段**：RPA 客户端从平台 chatList 最后一条气泡读取的时间（含买家和客服互动），用于在"人工处理中"判断是否真的"还在跟买家聊"。比 `ctx.updated_at` 更准——因为客服在平台回买家但没回调给本服务时，ctx 时间不会推进。
3. **10 分钟去抖**：人工处理中 10 分钟内的重复消息，**不发回复不发告警**，只把 Future 填空让 HTTP 不超时。超过 10 分钟则视为新一轮对话，重置 ACTIVE 走正常流程。
4. **category_id 双重优先级**：店铺配置里 `category_id` 既是向量检索的"分类共享 collection"key，又是 FAQ / 关键词 / 搪塞话术的归属键。管理后台是真实数据源，启动时一次性把"按 category_id+shop_id 缓存"加载到内存。
5. **`_handle_message` 与 listener 是双通道**：HTTP 走 `_handle_message`（同步等 Future），listener 走 `scheduler.enqueue`（异步）。两者最终都汇入 `_queues[shop_id]` 队列，调度器无差别消费。HTTP 路径多了一个"同步等 reply"环节——这是为了让前端调试页能拿到即时结果。
6. **`DebugContext` 双注册**：除了 `gateway._pending_debug_ctx[message_id]`，还通过 `register_debug_context(message_id, ctx)` 存到 module-level registry（`src/utils/trace.py`），这样异步子任务（dispatcher 链路）也能按 message_id 找回 context 写入步骤。API 处理完后两者都清理。
7. **置信度格式硬约束**：Prompt 强制 LLM 输出 `[CONFIDENCE: XX]`，`LLMResponse.raw_text` 解析该标记作为 confidence。LLM 不输出标记则 confidence=0 → 转人工。
8. **店铺动态兜底**：若 `shop_id` 不在静态配置中，系统会从 DB 构造最小 `ShopConfig(shop_id, platform, name=shop_id, category_id, confidence_threshold=85)`，让新店铺在还没纳入静态配置时也能跑通。
