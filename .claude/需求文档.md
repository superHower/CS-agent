三、接入层
3.1 RPA 真实数据格式
影刀 RPA 每次触发推送一个完整的会话数据包（新格式）：

json
{
    "shop_id": "tb_lamp_001",
    "history": [
        {
            "platform": "淘宝",
            "shop": "艾睿斯旗舰店",
            "buyer": "买家昵称",
            "product": "商品名或无",
            "chatList": [
                "2025-06-22 10:00:00 买家：你好",
                "2025-06-22 10:00:05 客服：欢迎光临",
                "2025-06-22 10:01:00 买家：这个产品还有货吗"
            ],
            "detail": "订单详情或无"
        }
    ],
    "message_id": "唯一ID（可选）"
}
取 history[-1] 为当前会话。

chatList 是气泡数组，包含时间、角色、消息内容。

product / detail 为“无”时统一处理为空字符串。

为兼容过渡期，旧格式（直接传 buyer_id, content, platform）继续支持，但内部全部转换为新格式的统一标准。

3.2 消息解析器 RpaParser
输入：RPA JSON

解析 chatList，提取完整对话历史（history 列表，每项 {role, content, time}）和最新买家消息（current_msg）。

无法提取买家消息时（纯系统消息），直接返回安抚话术 {"reply": "亲，稍等我查一下哈~", "escalated": true} 并终止处理。

提取 product 和 detail 填充到标准消息中。

3.3 标准化消息模型
python
class StandardMessage(BaseModel):
    shop_id: str
    platform: Platform          # 枚举
    buyer_id: str
    current_msg: str            # 最新买家消息
    history: list[DialogTurn]   # 完整对话历史
    product_name: str           # 商品名，空串表示无
    order_detail: str           # 订单详情，空串表示无
    message_id: str
    timestamp: datetime
    source: MessageSource.RPA
    raw_payload: dict
3.4 网关 RpaGateway
POST /api/message 路由。

去重：Redis Key msg_dedup:{shop_id}:{message_id}，TTL 24h。

内部维护 asyncio.Queue 和 message_id → asyncio.Future[(reply, escalated)] 映射。

30s 未回复 → 返回安抚话术并 escalated: true。

## 四、会话调度层

### 4.1 会话上下文
Redis `session:{shop_id}:{buyer_id}` 存储 `SessionContext`，包含完整历史、状态（ACTIVE/WAITING_HUMAN/ENDED），TTL 2h。

### 4.2 调度逻辑
1. 硬转人工关键词（投诉/差评等）→ 立即假回复 + 告警。
2. 状态为 WAITING_HUMAN → 安抚话术 + 告警，不自动回复。
3. 否则调用匹配决策层：
   - 返回可自动回复 → 发送真回复。
   - 返回需转人工或异常 → 发送搪塞话术 + 告警。

## 五、匹配决策层（核心重设计）

### 5.1 输入输出
```python
class MatchRequest(BaseModel):
    user_msg: str
    full_history: list[DialogTurn]
    shop_id: str

class MatchResult(BaseModel):
    reply: str
    source: str          # "faq_cache" | "intent_rag"
    confidence: float
    needs_escalation: bool
```

### 5.2 第一关：FAQ 精确缓存

- 对 `user_msg` 标准化（去标点、小写、合并空格）→ MD5 → 查 Redis `faq:{shop_id}:{hash}`。
- 命中 → 直接返回对应回复，`source="faq_cache"`, `confidence=100`。
- FAQ 可由管理后台维护，也支持从历史高频问题自动沉淀。

### 5.3 第二关：意图识别 + 检索 + 生成

当 FAQ 未命中时，进入统一的智能处理管道：

#### 步骤 2.1：LLM 意图识别（轻量 Prompt）

- **目标**：用极少 token 理解买家意图，抽取关键信息。
- **调用专用 `intent_prompt`**，要求 LLM 返回结构化 JSON（或简短文本）：
  ```
  分析买家问题，提取核心意图和检索查询词。
  回复格式：
  {
    "intent": "安装咨询 | 产品推荐 | 售后问题 | 物流询问 | 闲聊 | 其他",
    "entities": ["关键词1", "关键词2"],
    "rewrite_query": "改写后的用于检索的完整句子"
  }
  ```
- **超时控制**：必须使用轻量模型（如 gpt-3.5-turbo 或本地小模型），超时 2 秒，失败则降级为直接转人工。
- **避免重复调用**：此步骤产生的意图信息暂存，供后续生成时复用。

#### 步骤 2.2：查询改写与向量检索

- 利用意图识别返回的 `rewrite_query` 作为最终查询，并用 `entities` 补充权重。
- 对 `rewrite_query` 做本地 `bge-small-zh` 嵌入 → 查询 Qdrant `collection_{shop_id}`，返回 Top-3 相似文本片段。
- 知识库内容包含：
  - 所有产品的 `attributes` 自然语言描述（包含详细参数）。
  - 人工快捷短语（完整短语内容，作为文本块存入向量库）。
  - 通用知识（安装教程、发货政策、质保说明、售后处理规则等）。
  - 所有内容统一索引，不再单独做关键词匹配。

#### 步骤 2.3：LLM 生成回复与置信度

- 构建完整 Prompt，包含：
  - 固化 System Prompt（店铺身份、售后原则、产品基础知识）。
  - 对话历史（`full_history`）。
  - 意图识别结果（intent, entities）。
  - 检索到的知识片段（`knowledge_text`）。
  - 当前用户消息。
- 要求 LLM 生成自然回复，并在末尾附加 `[CONFIDENCE: 0-100]`。
- 若 LLM 调用超时（总超时 5s）或生成失败 → 转人工。
- 置信度阈值判断：
  - 若 `confidence >= shop_threshold`（默认75%）→ 自动发送。
  - 否则 → 转人工。

### 5.4 性能与降级策略

- **意图识别**：超时或失败 → 跳过意图，直接用原始用户消息做检索，然后尝试生成；若连续失败则转人工。
- **向量检索**：超时或 Qdrant 不可用 → 只使用固化 System Prompt 生成（置信度会很低，大概率转人工）。
- **总时间控制**：非 FAQ 路径总耗时上限 7 秒（含意图识别、检索、生成），超时则转人工。

## 六、LLM 推理层

- 统一接口，支持两种调用模式：
  - **意图识别模式**：低温度、JSON 输出、max_tokens 200。
  - **生成回复模式**：正常温度、文本输出、max_tokens 500。
- 可通过配置切换云端或本地模型（优先使用本地 Qwen2-7B 以减少延迟和成本）。
- 记录每次调用的 tokens、耗时，用于按店铺统计。

## 七、知识库构建策略

### 7.1 内容来源
1. **产品信息**：从 MySQL `products` 表取出，每款产品的 `attributes` 文本作为一个文档，向量化存入 Qdrant。
2. **快捷短语**：人工客服的标准话术（安装说明、优惠活动、售后流程等），以“问题-答案”对或独立段落形式存入 Qdrant，不再单独做倒排索引。
3. **通用知识**：公司文档（细节注意事项.md）中的规则、政策，整理成短段落存入 Qdrant。
4. **FAQ**：高频问答对除了缓存到 Redis，也存入 Qdrant 作为补充，提高语义检索的命中率。

### 7.2 同步机制
- 管理后台修改产品或快捷短语时，实时调用 `embed_and_upsert` 更新 Qdrant，支持增删改。
- 启动时全量校验向量库与 MySQL/文件内容的一致性。

待处理人工队列：展示 pending_alert:{shop_id} 中的会话列表，包含店铺、买家（脱敏）、触发原因、最后对话摘要。

快速处理：标记“已处理”或“转入售后”，从队列移除。

告警规则配置：管理必须转人工的关键词列表（如“投诉”“12315”），可自定义添加。

告警渠道测试：企业微信 Webhook 连接测试。

七、系统配置
店铺管理：增删改店铺，配置平台类型、店铺名称、置信度阈值、话术模板。

搪塞话术池管理：维护可随机选取的安抚语。

LLM 配置：模型供应商选择、API Key、超时时间、温度等，修改后即时生效。

数据归档策略：会话保存天数、自动清理规则。
