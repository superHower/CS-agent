## 管理后台数据库设计

基于最终架构（接入层 → 调度层 → 匹配决策层 [FAQ直达/意图识别+检索+LLM] → 动作执行层），MySQL 作为主存储负责：产品信息、知识库内容、配置管理、对话归档、操作日志等。

### 一、核心业务表

#### 1. 产品表 `products`

```sql
CREATE TABLE products (
    id          INT AUTO_INCREMENT PRIMARY KEY,
    shop_id     VARCHAR(50)  NOT NULL DEFAULT 'global' COMMENT '店铺ID，或全局共享时用global',
    model       VARCHAR(100) NOT NULL COMMENT '型号，全局唯一',
    attributes  TEXT         NOT NULL COMMENT '自然语言属性描述，包含所有参数变体',
    tags        VARCHAR(500) COMMENT '标签，逗号分隔，便于搜索',
    qdrant_sync TINYINT(1)   DEFAULT 1 COMMENT '是否已同步到Qdrant：1已同步, 0待同步, -1失败',
    created_at  DATETIME     DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_model_shop (model, shop_id),
    INDEX idx_tags (tags)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```
如果所有店铺完全共享产品，`shop_id` 恒为 `global`；如果未来需要店铺独立产品，保留此字段。

#### 2. 知识库内容表 `knowledge_entries`

存储快捷短语、通用知识文档等所有非产品知识文本，统一管理并同步到 Qdrant。

```sql
CREATE TABLE knowledge_entries (
    id          INT AUTO_INCREMENT PRIMARY KEY,
    shop_id     VARCHAR(50)  NOT NULL DEFAULT 'global',
    category    VARCHAR(50)  COMMENT '分类：shortcut, policy, faq_supplement, tutorial',
    code        VARCHAR(100) COMMENT '快捷短语的code，如安装、优惠；其他类型可为空',
    title       VARCHAR(200) COMMENT '标题，用于知识文档',
    content     TEXT         NOT NULL COMMENT '完整文本内容',
    status      TINYINT(1)   DEFAULT 1 COMMENT '1=已发布, 0=草稿, -1=已删除',
    qdrant_sync TINYINT(1)   DEFAULT 1 COMMENT '同步状态，同products',
    created_at  DATETIME     DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_shop_category (shop_id, category),
    INDEX idx_code (code)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```
快捷短语以 `category='shortcut'` 存储，`code` 用于追溯原标签，方便管理后台按 code 筛选；内容写入 Qdrant 进行语义检索。

#### 3. FAQ 表 `faq_entries`

```sql
CREATE TABLE faq_entries (
    id          INT AUTO_INCREMENT PRIMARY KEY,
    shop_id     VARCHAR(50)  NOT NULL DEFAULT 'global',
    question    TEXT         NOT NULL COMMENT '标准问题（用于计算hash）',
    answer      TEXT         NOT NULL COMMENT '标准回复',
    hash_key    VARCHAR(64)  NOT NULL COMMENT '问题标准化后的MD5/SHA256，用于Redis键',
    status      TINYINT(1)   DEFAULT 1 COMMENT '1=启用, 0=停用',
    created_at  DATETIME     DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_hash (hash_key, shop_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```
管理后台维护时自动生成 `hash_key` 并同步写入 Redis `faq:{shop_id}:{hash}`，同时也可选择将 FAQ 问答对作为知识条目存入 Qdrant 增加检索覆盖。

#### 4. 属性词典表 `attribute_dict`（可选）

```sql
CREATE TABLE attribute_dict (
    id          INT AUTO_INCREMENT PRIMARY KEY,
    keyword     VARCHAR(100) NOT NULL COMMENT '缩写/口语词',
    expansion   VARCHAR(200) NOT NULL COMMENT '正式全称或解释',
    category    VARCHAR(50)  COMMENT '分类',
    shop_id     VARCHAR(50)  DEFAULT 'global',
    UNIQUE KEY uk_keyword_shop (keyword, shop_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```
用于查询改写，将用户口语词（如“A款”）扩展为正式描述（“A款智能吸顶灯 XD-2401A”）。

### 二、会话与归档表

#### 5. 对话归档表 `conversation_archive`

```sql
CREATE TABLE conversation_archive (
    id            INT AUTO_INCREMENT PRIMARY KEY,
    shop_id       VARCHAR(50)  NOT NULL,
    buyer_id      VARCHAR(100) NOT NULL,
    session_id    VARCHAR(100) COMMENT '关联会话标识',
    summary       TEXT         COMMENT '对话摘要或提取的关键信息',
    full_history  JSON         COMMENT '完整对话历史 [{role, content, time}]',
    resolution    VARCHAR(50)  COMMENT '处理结果：auto_reply, escalated, human_closed',
    created_at    DATETIME     DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_shop_buyer (shop_id, buyer_id),
    INDEX idx_created (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```
会话结束时（Redis TTL过期或手动触发）异步写入。若需快速查询，可定期归档到独立存储。

### 三、配置与监控表

#### 6. 店铺配置表 `shop_config`

```sql
CREATE TABLE shop_config (
    shop_id             VARCHAR(50) PRIMARY KEY,
    platform            VARCHAR(20)  NOT NULL COMMENT 'taobao/pinduoduo/jd/douyin',
    shop_name           VARCHAR(100) COMMENT '店铺显示名称',
    confidence_threshold TINYINT UNSIGNED DEFAULT 75 COMMENT '置信度阈值，0-100',
    decoy_phrases       JSON         COMMENT '店铺专属搪塞话术列表，覆盖全局配置',
    is_active           TINYINT(1)   DEFAULT 1,
    created_at          DATETIME     DEFAULT CURRENT_TIMESTAMP,
    updated_at          DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```
替代之前静态 YAML 配置，支持后台热更新。

#### 7. LLM 配置表 `llm_config`

```sql
CREATE TABLE llm_config (
    id          INT AUTO_INCREMENT PRIMARY KEY,
    provider    VARCHAR(50)  NOT NULL COMMENT 'openai / qwen_local / doubao',
    model       VARCHAR(100) NOT NULL,
    api_base    VARCHAR(200) COMMENT 'API地址，本地模型时填写内网URL',
    api_key     VARCHAR(200) COMMENT 'API密钥，加密存储或从env读取',
    timeout     INT          DEFAULT 5 COMMENT '秒',
    is_active   TINYINT(1)   DEFAULT 1,
    created_at  DATETIME     DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```
支持多套模型，意图识别和生成可用不同配置。实际运行时通过环境变量注入密钥更安全，表内可留空。

#### 8. 告警配置表 `alert_config`

```sql
CREATE TABLE alert_config (
    id              INT AUTO_INCREMENT PRIMARY KEY,
    shop_id         VARCHAR(50)  DEFAULT 'global',
    webhook_url     VARCHAR(500) NOT NULL COMMENT '企业微信机器人Webhook',
    mention_users   VARCHAR(200) COMMENT '@的手机号或userid',
    is_active       TINYINT(1)   DEFAULT 1,
    created_at      DATETIME     DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

#### 9. 转人工关键词表 `escalation_keywords`

```sql
CREATE TABLE escalation_keywords (
    id       INT AUTO_INCREMENT PRIMARY KEY,
    keyword  VARCHAR(100) NOT NULL UNIQUE COMMENT '触发词，如投诉、12315',
    shop_id  VARCHAR(50)  DEFAULT 'global'
);
```
调度层每次从 DB 加载关键词表（可缓存至 Redis）。

#### 10. 搪塞话术池表 `decoy_phrases_pool`

```sql
CREATE TABLE decoy_phrases_pool (
    id      INT AUTO_INCREMENT PRIMARY KEY,
    shop_id VARCHAR(50)  DEFAULT 'global',
    phrase  TEXT         NOT NULL COMMENT '安抚话术'
);
```
全局默认 + 可覆盖的店铺级话术。

### 四、系统日志表

#### 11. 操作日志表 `admin_logs`

```sql
CREATE TABLE admin_logs (
    id          INT AUTO_INCREMENT PRIMARY KEY,
    user_id     VARCHAR(50)  COMMENT '操作管理员',
    action      VARCHAR(100) NOT NULL COMMENT '如 update_product, delete_faq',
    target      VARCHAR(200) COMMENT '操作对象标识',
    detail      TEXT         COMMENT '变更详情JSON',
    ip_address  VARCHAR(45),
    created_at  DATETIME     DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 12. 消息处理日志表 `message_logs`

```sql
CREATE TABLE message_logs (
    id              INT AUTO_INCREMENT PRIMARY KEY,
    shop_id         VARCHAR(50),
    buyer_id        VARCHAR(100),
    message_id      VARCHAR(100),
    user_msg        TEXT,
    match_source    VARCHAR(30) COMMENT 'faq_cache/intent_rag/fallback',
    reply           TEXT,
    confidence      FLOAT,
    elapsed_ms      INT,
    llm_tokens_in   INT,
    llm_tokens_out  INT,
    is_escalated    TINYINT(1),
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_shop_time (shop_id, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```
可抽样记录或异步批量写入，用于统计分析。

### 五、数据库选型说明

采用 **MySQL**（或 MariaDB）理由：
- 稳定的异步驱动（`aiomysql`/`asyncmy`）与 Python 生态匹配。
- 支持 JSON 字段，方便存对话历史和话术列表。
- 比 SQLite 更适合多店铺并发写入和未来扩展。
- 无需引入额外运维成本，本地单机部署即可。

---

### 六、数据流与一致性保证

- **Redis 缓存（FAQ）** 以 MySQL `faq_entries` 为源，修改时双写（MySQL + Redis），若 Redis 写入失败则标记后续重试。
- **Qdrant 向量** 以 `products` 和 `knowledge_entries` 为源，通过 `qdrant_sync` 字段标记同步状态，后台可触发全量或增量同步。
- **配置热更新**：修改 `shop_config`、`llm_config` 等表后，通过 Redis Pub/Sub 通知各服务实例重载内存配置。

---

此设计满足架构的完整功能需求，且保持简洁，方便运营通过后台界面直接维护，无需接触数据库。
